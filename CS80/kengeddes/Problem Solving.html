Problem Solving

The hard part of computer programming is not learning the rules of syntax or becoming familiar with fundamental programming features such as variables, expressions, etc. The hard part is developing an algorithm that solves the problem. This is also known as planning and is something you must do before actually writing the computer code in some programming language such as JavaScript.

Solving a problem differs from following someone else's solution to a problem, just as writing a story differs from reading a story. You need analytical skills, imagination, and perseverance.

It helps to look at a lot of examples. Unfortunately, a lot of the examples in tutorials and reference manuals are toy programs designed only to illustrate the particular statement or language feature being discussed. Typically, your homework assignments require you to do something more complex and challenging.

There are two approaches:

  1. Top-down structured design and programming is a powerful, tried and true method for solving even the most complex problems. Think of what needs to be done. Use the divide-and-conquer technique to break the problem down into smaller and smaller sub-problems.
  2. In the bottom-up approach you consider what tools you have to work with, so you don't end up reinventing the wheel. For example, if you need to sort some items and a sort method already exists, use the existing sort method rather than write your own. You also may wish to create a set of tools (that is, methods or functions) that perform useful tasks such as displaying an error message or getting input from the user.

In practice, you use a combination of both techniques.

 

Example of Algorithm Development

This example uses all the control structures: sequence, selection, and repetition.

Prior to writing low-level code for a program of any complexity, it is useful to first develop the algorithm. Otherwise, you may find that weeks of programming can save hours of planning.  : )  Develop the algorithm from the top-down by a process Niklaus Wirth calls stepwise refinement. Start with a birds-eye view - a single sentence that states what must be done to solve the problem. This is your level 0 algorithm. Add some detail to create the level 1 algorithm. Continue this way until the algorithm is sufficiently detailed that you can easily code it in your target programming language.

 

Number Guessing Game

Problem statement: Play the number guessing game where one person thinks of a number and another person tries to guess the number.

 

0:

The first person thinks of a number, invites a second person to try to guess it, and gives feedback such as "That's too high" or "That's too low" until the second person correctly guesses the number.

 

Now, lets break the problem down into one or more parts or sub-problems. Let's call the two players A and B.

1:

A thinks of a number between one and ten.
A invites B to play the game.

  1. A asks B for his guess.
  2. B makes a guess.
  3. A gives feedback: 'you got it', or 'too low', or 'too high'.

If the number has not been guessed,

  1. A asks B for his guess.
  2. B makes a guess.
  3. A gives feedback: 'you got it', or 'too low', or 'too high'.

If the number has not been guessed,

  1. A asks B for his guess.
  2. B makes a guess.
  3. A gives feedback: 'you got it', or 'too low', or 'too high'.

If the number has not been guessed,

  1. A asks B for his guess.
  2. and so on...

 

We notice there is a repeating pattern here. We could more concisely express our algorithm as follows:

2:

  1. A thinks of a number between one and ten.
  2. A invites B to play the game.
  3. repeat until the number is guessed
    1. A asks B for his guess.
    2. B makes a guess.
    3. A gives feedback: 'you got it', or 'too low', or 'too high'.

 

Let's let the computer be player A. The user will be player B. Also, we continue to refine the algorithm by choosing names for the data, using math symbols, etc.

3:

  1. Generate a random number, target, between 1 and 10.
  2. Invite the user to try to guess the number.
  3. Repeat until number is guessed:
    1. Get the guess from the user.
    2. If guess equals target, display "You got it!"
    3. Else if guess < target, display "Too low."
    4. Else display "Too high."

 

<- is a pseudocode symbol for assignment. The algorithm begins to look more and more like code as we get nearer and nearer to implementing the algorithm in our target language.

4:

target <- a random number between 1 and 10.
display "I am thinking of a number between 1 and 10. Try to guess what it is."
guessed <- false
repeat while not guessed
   guess <- value entered by  user
   if guess == target
      display "You got it!"
      guessed <- true
   else if guess < target
      display "Too low."
   else
      display "Too high."

 

If we know what our target language is, we can start to use its syntax as we continue to refine our algorithm. Suppose we decide to implement our algorithm in Alice.

5:

target.set(random number with minimum 1, maximum 10);
alice.say("I am thinking of a number between 1 and 10. Try to guess what it is.");
guessed.set(false);
while (!guessed) {
   guess.set(number dialog("What is your guess?"));
   if (guess == target) {
      alice.say("You got it!");
      guessed.set(true);
   } else if (guess < target) {
      alice.say("Too low!");
   } else {
      alice.say ("Too high!");
   }
}

By now our algorithm is sufficiently refined that coding it in our target language is easy.

Or, suppose we decide to implement our algorithm in JavaScript. Then our level 5 algorithm is

5:

var target <- random() * 10 + 1
write("I am thinking of a number between 1 and 10. Try to guess what it is.")
var guessed <- false
while (!guessed)
   var guess <- prompt("Enter a number between 1 and 10")
   if (guess == target) {
      write ("You got it!")
      guessed <- true
   } else if (guess < target)
      write ("Too low.")
   else
      write ("Too high.")
