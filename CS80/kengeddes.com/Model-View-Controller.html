Model-View-Controller

The Model-View-Controller (MVC) architecture clearly separates applications into three parts: data model, presentation, and program logic.

  • The model is the data plus any constraints on the data.
  • The view prepares and presents results to the user.
  • The controller controls the interaction between the user and the application.

The MVC reduces the coupling among the three parts of an application making it easier to develop and maintain. Originally developed for GUIs with Smalltalk, it was later discovered to be a valuable architecture for other applications, especially web apps. 1

 

MVC is an application of the software engineering principle known as separation of concerns, which basically stated implies that it is best to segregate different types of functionality within an application as much as possible.

  • The model encapsulates the application logic and state. It maintains and manipulates the application's state. It includes a database, DBMS, and database driver.
  • The view presents a UI. It includes HTML, images, scripts, taglibs, etc. It includes all of the modules presented directly to the user.
  • The controller handles user actions. It includes handling user requests, validating requests, obtaining request parameters, and reformating data for transaction processing.

Advantages of the MVC architecture are simplicity, independence, scalability, and developer specialization. 2

Web App Tiers and the MVC Pattern

Web App Tiers

Most web apps have a three-tiered architecture:

  1. The HTTP client (the front end) runs in the browser. Client-side technologies, such as HTML, CSS, JS, Ajax, and others, are used here. Also, XML and JSON which are data representation technologies, may appear in all three tiers.
  2. The middle tier is the business logic that runs on the HTTP server. Server-side technologies, such as PHP, JSP, ASP, or ColdFusion among others, are used here.
  3. The information tier (the back end) is the data store. This is often a RDBMS that is manipulated using SQL.

The client cannot directly access the information tier. Rather, it does so indirectly through the web server as follows:

  1. The client sends an HTTP GET or POST request to the web server.
  2. The web server handles this request by creating a SQL query which it sends to the database system.
  3. The database system returns the query results to the web server.
  4. The web server puts these results in a form the client can understand - some combination of HTML, XML or JSON, etc. which it then sends to the client.

Figure 1. Web App Tiers

As you can see, the client, the server, and the database each have a role to play and a set of technologies they support. It is through the communication and cooperation of these three distinct tiers that the web app does its magic.

Conclusion

With the MVC approach, we organize our work into three areas of responsibility. Initially, this involves more work and yields a more complex app. However, it makes the app much more flexible, robust, secure, and extensible.
